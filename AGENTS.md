## 1. 에이전트 기본 제약 (Behavior Constraints)

### ⚠️ 최우선 규칙 (MUST)

> **1. `main` / `dev` 에 절대 직접 푸시하지 않는다.**
> 직접 `git push origin main` / `dev` 금지. 변경은 항상 **토픽 브랜치 → PR → 리뷰** 경로로.
> 사용자가 명시적으로 "main에 바로 올려" 라고 해도, **먼저 확인하고** 이슈→PR 흐름을 권할 것.

> **2. 모든 변경은 `이슈 → PR` 과정을 거친다.**
> 사소한 수정이라도 해당 이슈 번호를 PR에 연결(`Closes #N`). 이슈 없는 PR은 먼저 이슈부터 만들 것을 권장.

> **3. 개발 논의가 나오면 가장 먼저 "이슈로 등록하자"고 추천한다.**
> 기능/버그/리팩터 안은 채팅에서만 끝내지 말고, **논의 시작 단계에서 GitHub 이슈로 옮길 것을 우선 제안**.
> 이슈가 추적·회고의 단일 소스가 됨.

### 작업 전
- 현재 브랜치·`git status` 먼저 확인. 기본 작업 브랜치는 `dev`.
- 어떤 변경을 할지 **짧게 계획을 말한 뒤** 진행.
- 새 기능/버그 이야기가 나오면 → **"이슈로 올릴까요?"** 가 반사적으로 나올 것.

### 변경/커밋/배포
- **커밋·푸시·배포는 명시적 지시가 있을 때만.** 허락 없이 원격 반영 금지.
- **`main`·`dev` 직접 push 금지** (위 ⚠️ 1번). 토픽 브랜치 → PR 만 허용.

### 커뮤니케이션
- 한국어로 응답. 간결하게.
- 불확실하면 추측하지 말고 **먼저 물어볼 것.**

## 2. 브랜치 / 워크플로우

```
이슈 등록 (#N)
   ↓
feat/<topic>   ← dev 에서 분기, 여기서만 커밋
   ↓
PR (Closes #N) → 리뷰
   ↓
dev   ← PR 병합 대상 (여기도 직접 push ❌, PR로만)
   ↓
main  ← 배포용(안정). dev→main 병합도 PR로
```

- **필수 흐름**: `이슈` → `feat/<topic>` 브랜치 → `PR` → 병합.
- 브랜치명: `feat/<topic>` · `fix/<topic>` · `docs/<topic>` (커밋 type과 일치).
- `main`·`dev` 모두 **직접 push 금지**, 오직 PR 병합으로만 갱신.
- PR 본문에 `Closes #N` 으로 이슈 연결.

---

## 3. 커밋 메시지 스타일

**Conventional Commits + 한국어.**

```
<type>(<scope>): <한국어 요약>
```

- **type** (필수): `feat` `fix` `docs` `style` `refactor` `perf` `test` `chore` `build` `ci`
- **scope** (선택): `api` · `chart` · `cache` · `ui` · `indicators` · `readme` 등
- **요약**: 한국어, 명사구 종결, **50자 이내** 권장. 끝에 마침표·`.` 금지.
- **body** (선택): "왜" 했는지 설명. 72자 줄바꿈.

예시:
```
feat(chart): 볼린저밴드 상하단 라벨 추가
fix(cache): 만료 직후 중복 업스트림 호출 수정
docs(readme): 라이브 데모 링크 추가
refactor(api): detail 라우트 응답 타입 정리
chore: 의존성 업데이트
```

> 과거 커밋(`README: …`)과 섞여 있으나, **이후 커밋은 위 규칙을 따른다.**

---

## 4. 이슈 등록 스타일

**제목** — 커밋과 동일한 접두어:
```
<type>: <한국어 요약>
```
예: `feat: 프론트 상단 버전 표시 + Changelog 뷰 추가`

**본문 템플릿** (아래 섹션 순서 준수):

```markdown
## 배경
- 왜 이 작업이 필요한지 (현재 상태 · 문제점)

## 작업 범위
### 1. <소주제>
- [ ] 세부 항목
### 2. <소주제>
- [ ] 세부 항목

## 체크리스트 (완료 기준)
- [ ] 검증 가능한 완료 조건

## 메모
- 참고 링크 · 후속 이슈 · 비고
```

- **라벨**: `enhancement` / `bug` / `documentation` / `question` 중 적절히 1개 필수.
- 큰 작업은 `작업 범위`를 번호 소섹션으로 나누고, 각 항목을 체크박스로.

---

## 5. PR 스타일

- 제목: 해당 브랜치의 대표 커밋 메시지 형식(`feat(...): …`).
- 본문: "무엇이 바뀌었는지" + "어떻게 테스트했는지" + 관련 이슈 번호(`Closes #N`).
- PR 본문 상단에 변경 요약 3~5줄.

---

## 6. CHANGELOG 규칙

> `CHANGELOG.md`가 아직 없으면 이 규칙에 따라 신규 생성한다.

- **모든 사용자 가시적 변경은 `CHANGELOG.md`에 기록.** (기능 추가 · 버그 수정 · 동작 변경)
- 형식: [Keep a Changelog](https://keepachangelog.com/ko/1.1.0/) — `Added` / `Changed` / `Fixed` / `Removed`.
- 버전: [Semantic Versioning](https://semver.org/lang/ko/) — 현재 `v0.1.0` 시작.
- PR에 changelog 항목이 없으면 머지 보류(셀프 체크).
